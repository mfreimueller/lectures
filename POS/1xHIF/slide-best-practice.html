Single Responsibility Principle

Each class should have one responsibility

  • responsibility = reason to change
  • Suppose a class printing PDF reports
  • Reasons to change:
    1. Content of report could change
    2. Format of report could change
  • 2 reasons = 2 responsibilities → split in two classes

Open-Closed Principle

Classes should be open for extension but closed for modification

  • Classes in libraries, when changed, require many programs to change
  • How to extend the StringBuilder class?
  • Inheritance (Java)
  • Extensions (Kotlin)

Dependency Inversion Principle

Depend upon abstractions, not concretes

  • Depending on interfaces (abstractions) not implementations
  • Example: Repositories in our projects

  • What are our benefits from using repositories?
    • Unit testing: Mocking
    • Swapping implementations without notice

What is a Pattern?

A design pattern is a reusable solution to a recurring design problem.
Gamma, E., Helm, R., Johnson R., & Vlissides, J. (1995). Design Patterns: Elements of Reusable Object-Oriented Software. Addison Wesley Longman, Inc.

Design patterns give programmers a way to talk about problems and solutions without talking about code.

Builder Pattern

Purpose: Creational pattern that abstracts instantiation processes

class UserBuilder {

  UserBuilder name(name) { this.name = name; return this; }
  UserBuilder email(email) { this.email = email; return this; }
  User build() { return new User(this.name, this.email); }

}

const user = new UserBuilder()
    .name("Marsha")
    .email("marsha@mellow.com")
    .build();

Repository Pattern

Purpose: Architectural pattern for data access abstraction

interface UserRepository {
  User findById(id);
  User save(user);
}

class UserService {

  UserService(UserRepository repository) {}

  User getUser(id) {
    return repository.findById(id); 
  }
}

Façade Pattern

Purpose: Provides a unified interface to a set of interfaces in a subsystem

Example: Services (in principle)

class OrderService {
  OrderService(
    OrderRepository orderRepo, 
    Logger logger, 
    PaymentService paymentService
  ) {}
  
  Order createOrder(CreateOrderCmd cmd) {
    var order = orderRepo.create(/* ... */);
    this.logger.log("Order created");
    return order;
  }
}

Strategy Pattern

Purpose: Behavioral pattern for defining interchangeable algorithms

interface PaymentStrategy {
  void pay(Long amount);
}

class CreditCardPayment implements PaymentStrategy {
  void pay(Long amount) { /* credit card logic */ }
}

class PayPalPayment implements PaymentStrategy {
  void pay(Long amount) { /* paypal logic */ }
}

YAGNI Principle

You Aren't Gonna Need It

principle from Extreme Programming (XP)

Functionality should only be added when necessary, not when anticipated:

  • Prevent over-engineering
  • Reduce code bloat
  • Current requirements > speculative future needs
